# 更新日志：混合检索 + 重排功能

> 版本：v1.3.0
> 日期：2026-06-16
> 功能：在召回阶段引入 BM25 词法检索，与原有向量检索融合；新增 Rerank 重排环节，提升整体检索准确率和召回率。

---

## 一、背景

### 原有方案（改造前）

知识库检索使用纯向量检索（Milvus + Embedding）：

```
用户查询 → Embedding 向量化 → Milvus 向量相似度搜索 → TOP 3 → 格式化上下文 → LLM
```

### 问题分析

| 问题 | 说明 |
|------|------|
| **召回率不足** | 向量检索擅长语义匹配，但对精确关键词（如产品型号、专有名词）匹配较弱 |
| **单一路径风险** | 只有一条检索路径，向量质量不佳时召回效果不稳定 |
| **排序粗糙** | TOP 3 完全依赖向量相似度，可能将真正相关的文档排在后面 |

```python

rag_vector_top_k: int = 10          # 向量检索召回数量
```

> - 如果想关掉某个环节，设置 `rag_enable_rerank = False` 即可

### 2.3 `app/services/bm25_service.py` — 新增 BM25 服务

**文件路径：** `app/services/bm25_service.py`（新建）

**核心职责：** 构建基于词法（关键词）匹配的检索引擎，作为向量检索的互补通道。

**工作流程：**

```
首次调用 search(query)
    ↓
构建语料池（只执行一次）
    ├─ collection.query_iterator 从 Milvus 取 rag_bm25_corpus_size 篇文档
    ├─ 对每篇文档做分词（中文逐字 / 英文按词）
    └─ 用 rank_bm25.BM25Okapi 构建索引，常驻内存
    ↓
对查询分词
    ↓
计算每篇文档的 BM25 得分
    ↓
TOP-K 取前若干篇
    ↓
Min-Max 归一化到 0~1
    ↓
返回 List[Tuple[Document, score]]
```

**关键设计点：**

| 设计点 | 选择 | 理由 |
|--------|------|------|
| 分词方式 | 中文逐字 + 英文按词 | 对中文而言，细粒度拆字比粗粒度分词更鲁棒；对英文自然按词拆分 |
| 语料构建 | 首次调用惰性构建 | 避免应用启动时阻塞；后续查询复用内存索引 |
| 得分归一化 | Top-K 范围内 Min-Max | 将 BM25 原始得分线性映射到 0~1，与向量检索得分可加权相加 |
| 强制重建 | `rebuild_corpus()` 方法 | 新增文档后可主动刷新语料池 |

**依赖库：** `rank-bm25 >= 0.2.2`

**公开接口：**

```python
from app.services.bm25_service import bm25_service

# 核心检索
results = bm25_service.search(query="查询文本", top_k=10)
# results: List[Tuple[Document, float_0~1]]

# 重建语料池（新增文档后）
bm25_service.rebuild_corpus()
```

### 2.4 `app/services/hybrid_search_service.py` — 新增混合检索服务

**文件路径：** `app/services/hybrid_search_service.py`（新建）

**核心职责：** 并行调用向量检索 + BM25 检索，加权融合后输出统一候选集。

**完整处理流程：**

```
输入: query
   ↓
[向量路径]
   ├─ query 向量化
   ├─ Milvus 原生 search (L2 metric)
   ├─ L2 距离 → 相似度: sim = 1 / (1 + L2)
   └─ Top-10 内部 Min-Max 归一化 → 0~1

[BM25 路径]
   ├─ query 分词
   ├─ BM25Okapi 计算得分
   └─ Top-10 内部 Min-Max 归一化 → 0~1

[融合]
   ├─ 以文档 _source / _file_name 为唯一键合并两路结果
   ├─ 对每个文档: final_score = 0.6 * vector_score + 0.4 * bm25_score
   │     (若某路未命中，给 0.05 保底分，避免仅单一路径的文档被过度压低)
   └─ 按 final_score 降序排序

[输出]
   └─ 取 TOP 20 → List[Document]
```

**关键设计点：**

- **去重策略**：用 `metadata._source` / `metadata._file_name` 作为文档唯一键，避免同一个文件的不同 chunk 被重复召回
- **单路径文档的保底处理**：如果一篇文档只在向量路径出现，BM25 分设为 0.05；反之亦然。这样既不会与双路径命中的文档竞争，也不会被完全剔除
- **权重可配置**：`rag_vector_weight` 与 `rag_bm25_weight` 在 config 中可调，经验值向量权重应略大于词法权重（0.6 / 0.4）

**公开接口：**

```python
from app.services.hybrid_search_service import hybrid_search_service

docs = hybrid_search_service.search("查询文本")
# docs: List[Document], 最多 rag_hybrid_top_k 篇（默认 20）
```

### 2.5 `app/services/rerank_service.py` — 新增加重排服务

**文件路径：** `app/services/rerank_service.py`（新建）

**核心职责：** 用更大更强的语义匹配模型对混合检索输出的候选文档进行二次排序，挑出真正最相关的 TOP 10。

**请求与响应结构：**

```
→ POST https://dashscope.aliyuncs.com/compatible-mode/v1/rerank
   Headers:
     Authorization: Bearer <DASHSCOPE_API_KEY>
     Content-Type: application/json
   Body:
     {
       "model": "rerank-v1",
       "query": "用户的查询问题",
       "documents": ["doc1 的文本内容", "doc2 的文本内容", ...],
       "top_n": 10
     }

← 响应:
   {
     "results": [
       {"index": 5, "relevance_score": 0.94},
       {"index": 2, "relevance_score": 0.87},
       ...
     ]
   }
```

**处理逻辑：**

1. 如果 `rag_enable_rerank = False`，直接返回原候选文档的前 TOP_K
2. 如果候选文档数 ≤ TOP_K，跳过重排直接返回
3. 否则调用 Rerank API，按返回的 `relevance_score` 重建排序
4. 如果 API 调用失败或超时，回退到原候选文档排序，记 warning 日志
5. 重排后的文档会在 `metadata._rerank_score` 字段写入重排得分，方便调试观察

**公开接口：**

```python
from app.services.rerank_service import rerank_service

# docs: List[Document]  来自混合检索的候选
final_docs = rerank_service.rerank(query="查询文本", documents=docs, top_k=10)
```

**注意事项：**

- 实际 DashScope Rerank 模型名可能不是 `rerank-v1`，请参考官方文档确认后修改 `config.rag_rerank_model`
- 需要与 `config.dashscope_api_key` 共用同一个 API Key
- 每篇文档文本限制为前 2000 字符，避免超长文本导致 API 报错

### 2.6 `app/tools/knowledge_tool.py` — 修改知识检索工具

**文件路径：** `app/tools/knowledge_tool.py`（修改）

**改造前后对比：**

| 步骤 | 改造前 | 改造后 |
|------|--------|--------|
| 1 | `vector_store.as_retriever(k=3)` | `hybrid_search_service.search(query)` → **TOP 20 候选** |
| 2 | — | `rerank_service.rerank(query, docs)` → **TOP 10 最终** |
| 3 | `format_docs(docs)` | `format_docs(docs)` |

**新的完整数据流向图：**

```
用户查询 (query)
   │
   ▼
┌──────────────────────────────────────────────────────┐
│  Step 1: 混合检索  hybrid_search_service.search()      │
│  ┌──────────────────────────────────────────────────┐ │
│  │  向量检索 TOP 10  (L2 → 相似度 → 归一化 0~1)     │ │
│  └──────────────────────────────────────────────────┘ │
│  ┌──────────────────────────────────────────────────┐ │
│  │  BM25 检索 TOP 10  (BM25 得分 → 归一化 0~1)      │ │
│  └──────────────────────────────────────────────────┘ │
│               ↓ 加权融合: 0.6 * v + 0.4 * b            │
│               ↓ 去重 + 排序                              │
│               →  输出 TOP 20 候选文档                   │
└──────────────────────────────────────────────────────┘
   │
   ▼
┌──────────────────────────────────────────────────────┐
│  Step 2: 重排  rerank_service.rerank()                │
│  ┌──────────────────────────────────────────────────┐ │
│  │  POST DashScope Rerank API                        │ │
│  │  query + documents[0..19]                         │ │
│  │  返回 relevance_score 排序结果                   │ │
│  └──────────────────────────────────────────────────┘ │
│               →  输出 TOP 10 最终文档                  │
└──────────────────────────────────────────────────────┘
   │
   ▼
format_docs() → 上下文文本
   ↓
返回给 Agent → LLM 回答问题
```

**降级 / 兜底逻辑：**

- 若混合检索返回空 → 回退到原来的纯向量 `vector_store_manager.similarity_search(k=10)`
- 若重排 API 失败 → 直接使用混合检索的前 TOP_K 篇
- 若重排后也为空 → 回退到混合检索的前 TOP_K 篇

**函数签名保持不变**（这很重要！）：`retrieve_knowledge(query: str) -> Tuple[str, List[Document]]`

因此，**对调用方（Agent 的 tool 使用逻辑）完全透明，不需要任何改动即可生效**。

## 三、效果预期

| 指标 | 改造前预期 | 改造后预期 | 改进来源 |
|------|-----------|-----------|---------|
| 召回率（相关文档被找到的概率） | 中等（仅向量语义匹配） | **高**（向量 + 关键词双通道覆盖） | BM25 词法匹配补充了精确关键词的覆盖 |
| 准确率（返回文档的相关性） | 中（TOP 3 全靠向量相似度） | **高**（重排模型二次过滤） | Rerank 用更强模型做细粒度相关判断 |
| 返回文档数 | 3 篇 | 10 篇 | 新流程允许更多候选并由重排精确过滤 |
| 单次检索耗时 | 快（向量单次查询） | 中等偏慢（新增 BM25 索引扫描 + Rerank API 调用） | 以时间换质量，可通过调参折中 |

## 四、常见调参指南

| 场景 | 推荐参数调整 |
|------|-------------|
| 知识库规模较大（> 500 篇） | 增大 `rag_bm25_corpus_size` 到 500~1000 |
| 知识库偏向专业术语 / 型号编号 | 提高 `rag_bm25_weight` 到 0.5 |
| 查询更偏向自然语言提问 | 提高 `rag_vector_weight` 到 0.7 |
| 担心 API 调用费用或延迟 | 设置 `rag_enable_rerank = False` |
| 重排后文档数过多，LLM 上下文装不下 | 减小 `rag_rerank_top_k` 到 3~5 |
| 新增了大量文档，BM25 语料池过时 | 调用 `bm25_service.rebuild_corpus()` |

## 七、回滚方案

如需回退到原有的纯向量检索方案，只需：

1. **临时降级**：将 `config.rag_enable_rerank = False`，并在 `knowledge_tool.py` 中改回使用 `vector_store_manager.similarity_search(query, k=3)`
2. **彻底回滚**：从版本控制中恢复 `knowledge_tool.py`，删除 3 个新建的 service 文件，在 `config.py` 中删除新增配置项
