# 多格式文件上传与索引支持文档

## 功能概述

本项目现已支持 **4 种文件格式**的自动上传与向量索引：

| 格式 | 扩展名 | 处理器类 | 关键特性 |
|------|--------|----------|----------|
| 纯文本 | `.txt` | `TextProcessor` | UTF-8/GBK 兼容编码，通用递归分割 |
| Markdown | `.md` `.markdown` | `MarkdownProcessor` | 按 H1/H2 标题分割，保留标题元数据 |
| **PDF** | `.pdf` | `PdfProcessor` | PyMuPDF 解析，表格识别，图片占位，代码块识别 |
| **Word** | `.docx` | `WordProcessor` | python-docx 解析，标题层级识别，表格转 Markdown |

> **注意**: Word 的旧格式 `.doc`（二进制格式）不在支持范围内，仅支持 Office 2007+ 的 `.docx` 格式。

## 架构设计：处理器工厂模式

### 目录结构

```
app/
├── processors/                    # 新增：文件格式处理器模块
│   ├── __init__.py                # 工厂函数（PROCESSOR_MAP 映射）
│   ├── base_processor.py          # 抽象基类 BaseProcessor
│   ├── text_processor.py          # TXT 处理器
│   ├── markdown_processor.py      # Markdown 处理器
│   ├── pdf_processor.py           # PDF 处理器（核心！）
│   └── word_processor.py          # Word 处理器
├── services/
│   ├── document_splitter_service.py   # 已改造：调用处理器工厂
│   └── vector_index_service.py        # 已改造：使用 process_file()
└── api/
    └── file.py                      # 已改造：扩展名验证从工厂获取
```

### 核心设计思路

```
文件上传 (.pdf)
    ↓
api/file.py: upload_file()
    → 验证扩展名（从 get_supported_extensions() 动态获取）
    → 保存到 ./uploads/
    ↓
vector_index_service.index_single_file(file_path)
    → 删除旧数据（按 _source 元数据匹配）
    → 调用 document_splitter_service.split_file(file_path)
    ↓
document_splitter_service.DocumentSplitterService.split_file()
    → 调用 processors.process_file(file_path, **kwargs)
    ↓
processors/__init__.py: process_file()
    → 根据扩展名从 PROCESSOR_MAP 查找处理器类
    → 实例化处理器 → 调用 processor.process(file_path)
    ↓
具体处理器 (如 PdfProcessor):
    1. read_file(file_path)  → 解析 PDF 原始内容
    2. split_content(content, file_path)  → 分片为 Document 列表
    3. 返回分片结果（带 _page / _block_type / _processor 等元数据）
    ↓
vector_store_manager.add_documents(documents)
    → 向量化 + 写入 Milvus
```

### 为什么用工厂模式？

| 优点 | 说明 |
|------|------|
| **可扩展** | 新增格式只需在 `app/processors/` 下加一个 `XxxProcessor` 类，并在 `PROCESSOR_MAP` 中注册一行，无需修改其他文件 |
| **统一接口** | 所有处理器都实现 `read_file()` + `split_content()` 方法，上层调用一致 |
| **向后兼容** | 保留了旧的 `split_document()` 接口（在 `document_splitter_service` 中），旧代码无需改动 |
| **元数据统一** | 基类 `BaseProcessor` 统一注入 `_source` / `_file_name` / `_extension` / `_processor` 元数据 |

## 各格式分片策略详解

### 1. TXT 处理器 (TextProcessor)

**读取方式**: `path.read_text()`，UTF-8 优先，失败回退 GBK

**分片策略**:
- 使用 LangChain 的 `RecursiveCharacterTextSplitter`
- `chunk_size = config.chunk_max_size` (默认 800 字符)
- `chunk_overlap = config.chunk_overlap` (默认 100 字符)
- 过小的分片（<200 字符）会被合并

**元数据**: `_block_type=paragraph`, `_source`, `_file_name`, `_extension`, `_processor`

---

### 2. Markdown 处理器 (MarkdownProcessor)

**读取方式**: `path.read_text(encoding='utf-8')`

**分片策略**（三阶段）：
1. **按标题分割**: `MarkdownHeaderTextSplitter` 按 `#` (H1) 和 `##` (H2) 分割
   - 每个分片的 metadata 自动注入 `h1` / `h2` 字段
2. **按大小二次分割**: 对大于 `chunk_size` 的块用 `RecursiveCharacterTextSplitter` 再次分割
3. **合并小分片**: < 300 字符的块合并到前一个分片

**元数据**: `h1`, `h2`, `_block_type`, `_source`, `_file_name`, `_extension`, `_processor`

---

### 3. PDF 处理器 (PdfProcessor) ⭐ 核心

**使用库**: `PyMuPDF (import fitz)`

**读取流程**:
```
打开 PDF 文档
    ↓
逐页处理 (page_num 从 1 开始):
    ├─ 步骤 A: 提取表格（优先，避免被文本重复提取）
    │   → page.find_tables() 检测表格
    │   → 提取为二维数组 → 转 Markdown 表格格式
    │
    ├─ 步骤 B: 提取图片信息
    │   → page.get_images(full=True) 获取图片列表
    │   → page.get_image_rects() 获取图片位置和尺寸
    │   → 生成 "[图片 N-M] (尺寸: WxH)" 文本占位符
    │
    └─ 步骤 C: 提取普通文本（按块分组，保留段落结构）
        → page.get_text("dict") 获取结构化文本
        → 过滤掉落在表格区域内的重复文本
        → 识别代码块（检测缩进 + 代码关键字）
```

**代码块识别逻辑**:
- 检测行缩进特征（连续空行 / 制表符开头）
- 检测代码关键字（`def` / `import` / `return` / `{` / `;` 等）
- 满足任一条件则包裹为 ````text ... ``` ` Markdown 代码块

**分片策略**:
1. **表格独立分片**: 每个表格作为完整的独立 Document（不截断）
2. **图片独立占位**: 每张图片的占位符作为独立块（可合并到相邻段落）
3. **代码块独立分片**: 每个识别出的代码块作为独立分片
4. **普通文本按页/按块累积**: 普通段落累积到 `chunk_size` 后新建分片
5. **二次分割**: 大于 `chunk_size` 的文本块使用 `RecursiveCharacterTextSplitter` 再次分割
6. **合并小分片**: < 250 字符的分片合并到前一个

**元数据** (每个分片都包含):
| 字段 | 类型 | 说明 |
|------|------|------|
| `_page` | int | 页码（从 1 开始） |
| `_block_type` | str | `paragraph` / `table` / `code` / `image` / `page` |
| `_has_table` | bool | 该页是否含表格 |
| `_has_image` | bool | 该页是否含图片 |
| `_has_code` | bool | 该页是否含代码块 |
| `_source` | str | 文件的 POSIX 路径 |
| `_file_name` | str | 文件名 |
| `_extension` | str | 扩展名 |
| `_processor` | str | 处理器类名（`PdfProcessor`） |

---

### 4. Word 处理器 (WordProcessor) ⭐ 核心

**使用库**: `python-docx (from docx import Document)`

**读取流程**:
```
打开 .docx 文档
    ↓
遍历所有段落 (doc.paragraphs):
    → 识别标题层级：检查 paragraph.style.name
        ├─ "Heading 1" / "标题 1" → H1
        ├─ "Heading 2" / "标题 2" → H2
        ├─ "Heading 3" / "标题 3" → H3
        └─ 其他样式 → 普通段落
    → 记录 paragraph_index 用于调试
    ↓
遍历所有表格 (doc.tables):
    → 逐行读取 row.cells
    → 清理单元格内的换行符
    → 组装为二维数组 → 转 Markdown 表格
```

**分片策略**:
1. **按标题层级分割**: 遇到 H1/H2/H3 标题时新建分片
   - 标题本身作为新分片的第一行（格式：`# 标题文本`）
   - metadata 记录 `h1` / `h2` / `h3`（当前上下文所属的各级标题）
2. **段落累积**: 段落文本累积到 `chunk_size`，超出则新建分片
3. **表格独立分片**: 每个表格作为完整的独立 Document（不截断）
   - 如果表格长度超过 `chunk_size`，用 `RecursiveCharacterTextSplitter` 二次分割
4. **合并小分片**: < 200 字符的分片合并到前一个

**元数据**:
| 字段 | 类型 | 说明 |
|------|------|------|
| `h1` | str | 当前分片所属的 H1 标题 |
| `h2` | str | 当前分片所属的 H2 标题 |
| `h3` | str | 当前分片所属的 H3 标题 |
| `_block_type` | str | `paragraph` / `table` |
| `_source` | str | 文件的 POSIX 路径 |
| `_file_name` | str | 文件名 |
| `_extension` | str | 扩展名 |
| `_processor` | str | 处理器类名（`WordProcessor`） |

## 前端文件类型校验详解 ⭐

### 为什么前端也需要白名单？

前端的文件类型校验有两个层面，缺一不可：

| 校验层 | 位置 | 作用 | 失败表现 |
|--------|------|------|---------|
| **accept 属性** | `static/index.html` | 浏览器文件选择对话框过滤 | 用户在文件选择器里看不到 PDF 文件 |
| **JS 二次校验** | `static/app.js:1104-1107` | 上传前 JavaScript 校验 | 弹出 "只支持上传 TXT / Markdown / PDF / DOCX 格式的文件" |

### accept 属性 (static/index.html:115)

```html
<!-- ✅ 正确配置：与 PROCESSOR_MAP 保持一致 -->
<input type="file" id="fileInput" accept=".txt,.md,.markdown,.pdf,.docx" style="display: none;">
```

- `accept` 是 HTML5 文件输入的原生属性，**由浏览器执行过滤**
- 值为逗号分隔的扩展名列表（注意每个扩展名前都需要点号）
- 浏览器会**灰掉**不在白名单中的文件，用户根本无法选中

### JavaScript 二次校验 (static/app.js:1104-1107)

```javascript
validateFileType(file) {
    const fileName = file.name.toLowerCase();
    // ✅ 与 accept 属性、后端 PROCESSOR_MAP 保持一致
    const allowedExtensions = ['.txt', '.md', '.markdown', '.pdf', '.docx'];
    return allowedExtensions.some(ext => fileName.endsWith(ext));
}
```

- `accept` 属性只是**浏览器侧的提示**，可以通过浏览器开发者工具或拖拽绕过
- `validateFileType()` 在 `handleFileSelect()` 和 `uploadFile()` 中被调用两次
- 双重校验确保**即使用户绕过文件选择对话框，也无法实际上传不支持的格式**

### 与后端的对应关系

```
前端 accept + allowedExtensions          后端 PROCESSOR_MAP
=========================                =======================
  .txt                                    ↔  ".txt"  → TextProcessor
  .md / .markdown                         ↔  ".md" / ".markdown" → MarkdownProcessor
  .pdf                                    ↔  ".pdf"  → PdfProcessor
  .docx                                   ↔  ".docx" → WordProcessor
```

> ⚠️ **历史问题记录**: PDF 和 DOCX 的处理器早在架构重构时就已完成，但前端 `accept` 和 `allowedExtensions` 未同步更新，导致用户无法上传 PDF 文件。修复时只需修改前端两个位置（参考上面的"修改清单"第 6、7 项）。

```python
    def read_file(self, file_path: str):
        # 用 BeautifulSoup / lxml 解析 HTML
        ...
        return structured_content

    def split_content(self, content, file_path: str):
        # 自定义分片逻辑
        ...
        return documents
```

### 步骤 2: 在工厂中注册
```python
# app/processors/__init__.py
from app.processors.html_processor import HtmlProcessor

PROCESSOR_MAP = {
    ...
    ".html": HtmlProcessor,  # 新增这一行
    ".htm": HtmlProcessor,   # 新增这一行
}
```

### 步骤 3: 更新前端白名单 ⚠️

```html
<!-- static/index.html: 增加 .html/.htm 到 accept 属性 -->
<input type="file" id="fileInput" accept=".txt,.md,.markdown,.pdf,.docx,.html,.htm" style="display: none;">
```

```javascript
// static/app.js: 增加 .html/.htm 到 allowedExtensions 数组
validateFileType(file) {
    const fileName = file.name.toLowerCase();
    const allowedExtensions = ['.txt', '.md', '.markdown', '.pdf', '.docx', '.html', '.htm'];
    return allowedExtensions.some(ext => fileName.endsWith(ext));
}
```

### 步骤 4: 完成

此时：
- API 层的 `_get_allowed_extensions()` 会自动包含 `.html` / `.htm`
- Service 层会自动调用 `HtmlProcessor`
- 上传接口会自动接受 HTML 文件
- 前端文件选择对话框也会显示 HTML 文件可选

> 💡 **关键提示**: 步骤 3 是最容易遗漏的！后端支持新格式但前端 `accept` 没更新 = 用户仍然选不到该文件。务必保持 `PROCESSOR_MAP` ↔ `accept` ↔ `allowedExtensions` 三者一致。

## 常见问题

### Q1: 上传 PDF 后提示 "PyMuPDF 未安装"
A: 执行 `uv add pymupdf` 或 `pip install pymupdf` 安装依赖。

### Q2: 旧的 `.doc` 文件无法上传？
A: `.doc` 是 Word 97-2003 的二进制格式，`python-docx` 不支持。
   解决方案：先用 Word / LibreOffice 将 `.doc` 另存为 `.docx`。

### Q3: PDF 中的图片只显示 "[图片 N-M]"，看不到图片内容？
A: 这是设计行为。向量数据库存储的是**文本描述**，图片本身无法向量化。
   `[图片 N-M] (尺寸: WxH)` 作为位置标记，让 LLM 知道"这里有一张图片"。
   如果需要图片内容识别，需要额外集成 OCR 服务（如 PaddleOCR）。

### Q4: 分片的 chunk_size 可以调整吗？
A: 可以。有三种方式：
   1. 修改 `app/config.py` 的 `chunk_max_size` / `chunk_overlap`（全局生效）
   2. 创建处理器时传参：`PdfProcessor(chunk_size=1200, chunk_overlap=200)`
   3. 创建 `DocumentSplitterService` 时传参：`DocumentSplitterService(chunk_size=1200)`

### Q5: 如何调试某个文件的分片结果？
A: 运行以下 Python 脚本：
```python
from app.processors import process_file

docs = process_file("path/to/your/file.pdf")
for i, doc in enumerate(docs):
    print(f"--- 分片 {i+1} (page {doc.metadata.get('_page')}, "
          f"{doc.metadata.get('_block_type')}, {len(doc.page_content)} 字符) ---")
    print(doc.page_content[:200] + "...")
    print()
```

### Q6: 我在前端点击上传文件时，选不到 PDF？
A: 这是典型的**前端白名单遗漏**问题，不是后端缺陷。请按以下顺序排查：

1. **检查文件选择对话框**：打开 `static/index.html`，搜索 `accept=`，确认值中包含 `.pdf`
2. **检查 JS 校验函数**：打开 `static/app.js`，搜索 `allowedExtensions`，确认数组中包含 `'.pdf'`
3. **检查错误提示**：如果有 "不支持的文件格式" 提示，搜索 `showNotification` 确认提示内容是否最新
4. **刷新浏览器缓存**：修改前端文件后需要按 Ctrl+F5（强制刷新），浏览器可能缓存旧的 HTML/JS

如果以上三处都正确但仍然无法上传，再检查后端是否正常启动并查看 `app/processors/pdf_processor.py` 是否可 import（即 PyMuPDF 是否已安装）。
