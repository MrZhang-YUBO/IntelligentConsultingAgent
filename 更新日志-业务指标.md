## 三、逐个文件详细说明（把你教明白）

### 3.1 `app/core/metrics.py`（核心新增文件）

这是整个功能的"大脑"，包含 6 个关键部分：

#### 1) 依赖导入（带兜底）
```python
try:
    from prometheus_client import (
        CONTENT_TYPE_LATEST, CollectorRegistry,
        Counter, Gauge, Histogram, generate_latest,
    )
    PROMETHEUS_AVAILABLE = True
except ImportError:
    PROMETHEUS_AVAILABLE = False  # 没装也不崩
```

> 💡 **为什么要 try-except 兜底？**
> 让项目在缺依赖时仍然可以正常启动（降级运行），不阻断开发。生产环境必须安装。

#### 2) 全局 CollectorRegistry
```python
REGISTRY = CollectorRegistry()   # 独立的注册表，避免和其他进程冲突
```

> 💡 **为什么不用默认的 REGISTRY？**
> 默认注册表是全局共享的。当你有多个 Python 进程（例如 pytest 多进程）运行时，默认注册表会出现重复注册的问题。显式创建一个独立注册表更干净。

#### 3) 定义的 6 个指标

| 指标名 | 类型 | Labels | 一句话说明 |
|--------|------|--------|----------|
| `http_request_count` | Counter | `method, path, status_code` | 累计请求数（按路径+状态码拆分） |
| `http_request_latency_seconds` | Histogram | `method, path` | 请求耗时（秒，默认分桶 0.005~10s） |
| `http_request_in_progress` | Gauge | `method, path` | 当前正在处理的请求数（并发数） |
| `http_error_count` | Counter | `method, path, status_code` | 4xx/5xx 错误数 |
| `service_request_latency_seconds` | Histogram | `service, method` | 业务层函数耗时（如 `rag.query`） |
| `service_error_count` | Counter | `service, method` | 业务层函数错误数 |

> 💡 **三种 Prometheus 类型的区别：**
> - **Counter（计数器）**：只增不减，用于"总数"类统计。如请求总数、错误总数。Prometheus 用 `rate()` / `irate()` 计算增速。
> - **Gauge（仪表盘）**：可增可减，反映"当前快照值"。如连接数、内存使用、正在处理的请求数。
> - **Histogram（直方图）**：按预设"桶"统计样本分布，用于计算 P50 / P95 / P99 延迟。典型用法：`histogram_quantile(0.95, rate(http_request_latency_seconds[5m]))`。

#### 4) FastAPI Middleware — `MetricsMiddleware`

```python
class MetricsMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        method = request.method
        path = _resolve_path(request)   # 归一化路径
        start_time = time.perf_counter()

        HTTP_REQUEST_IN_PROGRESS.labels(...).inc()
        try:
            response = await call_next(request)
            status_code = response.status_code
            return response
        except Exception:
            status_code = 500
            raise
        finally:
            elapsed = time.perf_counter() - start_time
            HTTP_REQUEST_IN_PROGRESS.labels(...).dec()
            HTTP_REQUEST_LATENCY.labels(...).observe(elapsed)   # ← 关键！
            HTTP_REQUEST_COUNT.labels(...).inc()
            if status_code >= 400:
                HTTP_ERROR_COUNT.labels(...).inc()
```

> 💡 **关键实现细节：**
> - `time.perf_counter()` 比 `time.time()` **精度更高**，且不受系统时间回拨影响，是测量耗时的标准选择。
> - `try/except/finally` 三件套：**即使 handler 抛出异常**，也能正确记录状态码 500 和耗时，保证指标不丢。
> - `_resolve_path(request)` 把真实路径 `/api/chat/session/123` 匹配成已注册的路由模板 `/api/chat/session/{session_id}`，**防止 label cardinality 爆炸**（否则每个不同的 session_id 都会生成新的 label，撑爆 Prometheus 内存）。

#### 5) 业务层装饰器 — `@timed_metric(service="rag", method="query")`

```python
def timed_metric(service, method):
    def decorator(func):
        async def async_wrapper(*args, **kwargs):
            start = time.perf_counter()
            try:
                return await func(*args, **kwargs)
            except Exception:
                SERVICE_ERROR_COUNT.labels(...).inc()
                raise
            finally:
                SERVICE_LATENCY.labels(...).observe(
                    time.perf_counter() - start
                )
        # 同步函数有另外一份 sync_wrapper
        if asyncio.iscoroutinefunction(func):
            return async_wrapper
        return sync_wrapper
    return decorator
```

> 💡 **为什么要分 sync / async 两种 wrapper？**
> FastAPI 的路由 handler 和业务函数可以是 `async def` 也可以是 `def`。如果把一个同步函数包在 `async def` wrapper 里，调用方需要 `await`，会破坏签名一致性。用 `asyncio.iscoroutinefunction()` 自动判断，让装饰器对两种写法都透明兼容。

#### 6) `/metrics` 响应生成
```python
def get_metrics_response_bytes():
    data = generate_latest(REGISTRY)
    return data, CONTENT_TYPE_LATEST
```
`CONTENT_TYPE_LATEST` 是 prometheus_client 提供的标准 MIME 类型：`text/plain; version=0.0.4; charset=utf-8`。Prometheus 抓取时会识别它。

新增两行配置（在原有的 Prometheus 配置 section 之后）：

```python
# 业务指标（本服务自己暴露 /metrics）
metrics_enabled: bool = True
```

> 💡 **为什么需要 `metrics_enabled` 开关？**
> - 单元测试时不需要真实的指标统计，设为 `False` 可以跳过中间件。
> - 多环境部署：开发环境可以关掉 `/metrics` 端点（或改路径），避免被扫描器探测。
> - `metrics_path` 允许改路径，应对反爬虫 / 安全策略（例如改成 `/internal/metrics` + Nginx 鉴权）。

### 3.3 `app/main.py`（应用入口改动）

#### 改动 1：导入
```python
from fastapi.responses import FileResponse, Response
from app.core.metrics import MetricsMiddleware, get_metrics_response_bytes
```

#### 改动 2：注册中间件
```python
# 配置 Prometheus 指标中间件（必须在 CORS 之后注册，先执行外层）
if config.metrics_enabled:
    app.add_middleware(MetricsMiddleware)
    logger.info(f"[Metrics] 已启用指标中间件，暴露路径: {config.metrics_path}")
```

> 💡 **Middleware 注册顺序很重要！**
> FastAPI 的 middleware 是**后进先出**（LIFO）的栈结构。
> - 请求进来：CORS → Metrics → 路由
> - 响应回去：路由 → Metrics → CORS
> 我们希望 Metrics 能测到 **包括 CORS 处理在内的整个请求**，所以让 Metrics 在 CORS 之后注册（即 Metrics 包裹在最外层）。

#### 改动 3：新增 `/metrics` 路由
```python
if config.metrics_enabled:
    @app.get(config.metrics_path, include_in_schema=True, tags=["监控"])
    async def metrics():
        content, content_type = get_metrics_response_bytes()
        return Response(content=content, media_type=content_type)
```

> 💡 **为什么用 `Response` 而不是 `JSONResponse`？**
> Prometheus 的指标格式是**纯文本**（不是 JSON），结构像这样：
> ```
> # HELP http_request_count Total HTTP requests
> # TYPE http_request_count counter
> http_request_count{method="GET",path="/health",status_code="200"} 42.0
> http_request_count{method="POST",path="/api/chat",status_code="200"} 17.0
> ```
> 用 `Response(media_type="text/plain; ...")` 可以正确输出这种格式。

> 对任何你关心耗时的函数，都可以加装饰器：
> ```python

## 四、指标与 PromQL 查询示例

### 4.1 HTTP 层

| 需求 | PromQL |
|------|--------|
| QPS（近 5 分钟速率） | `rate(http_request_count[5m])` |
| 按路径分的 QPS | `sum by (path) (rate(http_request_count[5m]))` |
| P95 延迟（5 分钟窗口） | `histogram_quantile(0.95, rate(http_request_latency_seconds[5m]))` |
| P99 延迟 | `histogram_quantile(0.99, rate(http_request_latency_seconds[5m]))` |
| 错误率 | `sum(rate(http_error_count[5m])) / sum(rate(http_request_count[5m]))` |
| 并发请求数 | `http_request_in_progress` |

### 4.2 业务层

| 需求 | PromQL |
|------|--------|
| RAG query 函数平均耗时 | `rate(service_request_latency_seconds_sum{service="rag",method="query"}[5m]) / rate(service_request_latency_seconds_count{service="rag",method="query"}[5m])` |
| RAG 流式对话 P95 耗时 | `histogram_quantile(0.95, rate(service_request_latency_seconds{service="rag",method="query_stream"}[5m]))` |

## 七、注意事项 & 最佳实践

1. **Cardinality（基数）控制**：永远不要把用户 ID、请求 ID 等**高基数**数据放进 label。一个 label 如果有 1000 种不同值，Prometheus 会存 1000 条独立时序，严重消耗内存。本次用 `_resolve_path()` 就是为了解决这个问题。

2. **Histogram 分桶**：默认分桶 `[0.005, 0.01, ..., 10.0]` 秒，对 LLM 对话场景（通常 1~10 秒）覆盖足够。如果你有更快的 API（如纯向量检索 < 100ms），可以按需自定义 `buckets` 参数。

3. **安全**：`/metrics` 端点在生产环境建议：
   - 在 Nginx / 反向代理层限制来源 IP，只允许 Prometheus server 访问
   - 或改路径为 `/internal/metrics`
   - 或加简单的 token 鉴权（`Authorization: Bearer <metrics-token>`）

4. **装饰器不会干扰返回值**：`@timed_metric` 用 `functools.wraps(func)` 保留原函数签名、文档字符串和返回类型注解，FastAPI 不会感知到装饰器存在。

5. **和现有 loguru 日志不冲突**：日志是"事件流"（离散文本），指标是"时序数据"（可聚合）。两者互为补充：指标告诉你"有问题"，日志告诉你"具体是什么问题"。
