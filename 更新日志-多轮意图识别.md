# 更新日志：多轮意图识别

> 版本：v1.5.0
> 日期：2026-06-24
> 功能：在 RAG 对话的 `query` / `query_stream` 中新增**多轮意图识别**阶段，识别结构化意图（支持多意图、复杂意图、长上下文、跨轮衔接），结果注入 Agent 系统提示词并暴露给前端。

---

## 一、背景与目标

### 1.1 之前的问题

`app/services/rag_agent_service.py` 的 `query` / `query_stream` 是「纯 RAG 对话」：用户问什么，Agent 拿全部历史 + 工具直接回答，**没有任何意图理解环节**。

| 问题 | 说明 |
|------|------|
| **工具选择被动** | Agent 不会主动判断该用哪个工具，多意图/复杂问题容易答偏或漏答 |
| **跨轮衔接弱** | 长对话中用户用代词（"它"/"这个"）追问时，缺少意图衔接与指代消解 |
| **多意图无感知** | 一句话含多个诉求（"查A并和去年对比"）时，无法识别并分别处理 |
| **复杂意图无标注** | 条件/依赖型问题（"如果CPU超80%就查告警"）无法结构化 |
| **长上下文风险** | 若直接把全部历史塞给识别模型，对话越长越慢越贵 |
| **可观测性弱** | 前端无法看到"系统理解到你在做什么" |

### 1.2 改造目标

- ✅ 每轮用快速模型识别**结构化意图**（Pydantic 模型）
- ✅ 支持**多意图**（主意图 + 副意图）、**复杂意图**（条件/依赖/多步）
- ✅ **多轮衔接**：结合上一轮意图判断切换/延续、指代消解
- ✅ **长上下文**：识别只接收有界上下文，与对话总长度无关
- ✅ 结果**注入 Agent 系统提示词**增强工具选择，并**通过 SSE + 会话历史接口暴露前端**
- ✅ **前端意图卡片**：在助手消息上方展示意图类型、置信度、实体、多意图/复杂标签，可展开查看详情
- ✅ 失败兜底，绝不中断主对话

## 二、整体架构

### 2.1 设计思想：复刻 SummaryAgent 模式

项目里已有一个成熟的「辅助 Agent」模式——`app/agent/summary_agent.py`（对话总结）。它的特征是：

- 独立的轻量模型类 + 模块级单例
- 在 `query` / `query_stream` 中、`_initialize_agent()` 之后被 `await` 调用
- 失败返回兜底文本，不中断主流程

**意图识别完全复刻这个模式**，新增 `app/agent/intent_agent.py`，在 `query` / `query_stream` 中紧跟在 `_summarize_and_update` 之后调用。这样改动集中、风格统一、风险可控。

### 2.2 调用时序

```
query / query_stream
   │
   ├─ await _initialize_agent()              # 初始化 Agent + 工具
   ├─ await _summarize_and_update(session_id)# 已有：先压缩历史
   ├─ await _recognize_intent(question, ...) # 【新增】意图识别
   │      └─ 读最近N条消息 + 已有总结 + 最近2条意图  ← 有界上下文
   │      └─ intent_agent.recognize(...) → IntentRecognitionResult
   │      └─ intent_tracker.record(session_id, result)  ← 记录轨迹
   │      └─ (流式) yield {"type":"intent","data":...}  ← 吐给前端
   ├─ system_prompt = _build_system_prompt(intent_context)  # 【改】按本轮意图重建提示词
   └─ agent.ainvoke / astream(...)
```

### 2.3 数据流向图

```
                  ┌─────────────────────────────────┐
   会话历史 ──────▶│  _read_checkpoint_messages       │
  (MemorySaver)   └────────────┬────────────────────┘
                                │ 取最近 N 条
                                ▼
   已有总结 ──────────▶ SummaryAgent.extract_existing_summary
                                │
   最近2条意图 ───────▶ intent_tracker.get_recent(k=2)
                                │
                                ▼
                    ┌───────────────────────────┐
                    │  intent_agent.recognize    │  qwen-turbo
                    │  with_structured_output    │  (有界输入)
                    └─────────────┬─────────────┘
                                  │ IntentRecognitionResult
                   ┌──────────────┼──────────────┐
                   ▼              ▼              ▼
            intent_tracker   format_for_prompt  to_dict
              .record()       → 注入系统提示词   → SSE/历史接口
```

## 四、意图分类体系（8 类）

映射到项目**已有的工具/能力**，确保识别出的意图有实际落脚点：

| IntentType | 含义 | 映射工具/能力 |
|---|---|---|
| `knowledge_qa` | 知识库问答 | `retrieve_knowledge` |
| `document_op` | 文档上传/列表/删除 | `/api/upload`、`/api/index_directory` |
| `aiops_diagnose` | 运维诊断/告警排查 | `query_prometheus_alerts` + MCP(cls/monitor) |
| `comparison` | 对比分析 | `retrieve_knowledge`（多实体） |
| `multi_step_task` | 多步复杂任务 | 多工具组合 |
| `clarification` | 澄清/追问（依赖上文） | 承接上一轮意图 |
| `chitchat` | 闲聊/问候 | 无需工具 |
| `unknown` | 无法判定 | — |

> 💡 **为什么是这 8 类？** 它们覆盖了当前 Agent 能做的所有事（查知识、传文档、查告警、对比、多步、追问、闲聊），且每类都能映射到具体工具。新增能力时只需在 `IntentType` 加一个枚举值 + 在系统提示词的分类表里加一行说明。

## 五、逐文件详解

### 5.1 `app/agent/intent_agent.py`（核心新增）

整个文件分三部分：**数据模型**、**IntentAgent**、**IntentTracker**。

#### ① 数据模型（Pydantic）

```python
class IntentType(str, Enum):       # 8 类，继承 str 便于 JSON 序列化
    KNOWLEDGE_QA = "knowledge_qa"
    ...

class SubIntent(BaseModel):         # 单个子意图
    intent_type: IntentType
    description: str                # 想做什么
    entities: list[str]             # 关键实体
    suggested_tools: list[str]      # 建议工具
    depends_on: str | None          # 复杂意图：依赖上文哪一轮/哪个意图

class IntentRecognitionResult(BaseModel):   # 一轮识别的完整结果
    primary_intent: SubIntent
    secondary_intents: list[SubIntent] = []  # 多意图时的副意图
    is_multi_intent: bool = False
    is_complex: bool = False                 # 条件/依赖/多步
    context_references: list[str] = []       # 引用了上文哪些内容（代词消解）
    intent_shift: bool = False               # 相对上一轮是否切换话题
    confidence: float                        # 0~1
    reasoning: str                           # 一句话理由

    def format_for_prompt(self) -> str: ...  # 生成注入系统提示词的文本块
    def to_dict(self) -> dict: ...           # 序列化给 SSE/JSON（model_dump(mode="json")）
```

> 💡 **`format_for_prompt()` 输出长这样**（注入 Agent 系统提示词末尾）：
> ```
> 【本轮意图识别（供你参考以选择工具、处理多意图/复杂意图）】
> 主意图: knowledge_qa（询问概念）| 置信度: 0.90
>   实体: 向量数据库
>   建议工具: retrieve_knowledge
> 多意图: 是（共 2 个）
>   - 子意图2: comparison（和去年对比）— 依赖: 上文产品A
> 复杂意图: 否 | 意图切换: 否
> 引用上文: 产品A
> 理由: 用户问概念并要求对比
> ```

#### ② IntentAgent 类

镜像 `SummaryAgent` 的结构：

| 成员 | 作用 |
|------|------|
| `__init__` | 建 `ChatQwen(model=config.intent_model, temperature=0, streaming=False)`——**温度 0** 保证意图识别的确定性 |
| `_build_system_prompt()` | 静态方法，含分类表 + 识别要求（多意图/复杂/多轮/实体/工具/置信度） |
| `_format_history(messages)` | 格式化最近对话，**每条截断 1500 字符**（与 SummaryAgent 一致），跳过 SystemMessage |
| `_format_previous_intents(intents)` | 把最近意图紧凑序列化，供判断 shift/依赖 |
| `_build_user_prompt(...)` | 拼装本轮输入：当前问题 + 已有总结 + 最近对话 + 最近意图 |
| `_unknown_result(query)` | 兜底结果（unknown），保证不中断 |
| `_parse_json(text)` | 从模型输出提取 JSON（兼容 ```` ```json ```` 代码块包裹） |
| `async recognize(...)` | **核心方法**，见下 |

`recognize` 的**三级容错**（关键设计）：

```
第1优先：model.with_structured_output(IntentRecognitionResult)   ← 复刻 planner 用法
   │ 失败(任何异常)
   ▼
第2优先：直接 ainvoke 让模型输出 JSON → _parse_json → model_validate
   │ 失败
   ▼
第3优先：返回 _unknown_result()   ← 绝不抛异常
```

> 💡 **为什么要三级容错？** `with_structured_output` 对「嵌套模型 + 枚举」在某些模型/版本上可能不稳定。第一级失败时退到「让模型直接吐 JSON 再解析」，再失败就返回 unknown。**最坏情况是对话少了意图增强，但绝不会崩**——这与 SummaryAgent「总结失败返回兜底文本」的理念完全一致。

> 💡 **为什么用 `with_structured_output` 而不是手写 JSON？** 它底层用 function-calling/tool-schema，模型被约束成必须按 Pydantic schema 输出，比"请输出 JSON"更可靠。`app/agent/aiops/planner.py` 已经这么用（`llm.with_structured_output(Plan)`），我们保持一致。

#### ③ IntentTracker 类

```python
class IntentTracker:
    # dict[session_id -> deque(maxlen=config.intent_history_size)]
    def record(session_id, result)        # 记录一轮
    def get_recent(session_id, k=2)       # 取最近 k 条（供下一轮 shift 判断）
    def get_all(session_id)               # 取全部轨迹（供会话历史接口）
    def clear(session_id)                 # 清空（与 clear_session 联动）
```

> 💡 **为什么用 `deque(maxlen=N)`？** 自动淘汰最旧意图，无需手动 trim。`maxlen=config.intent_history_size`（默认 10），即每个会话最多留 10 轮意图。

> 💡 **为什么是内存态？** 与 `MemorySaver`（对话记忆）同生命周期——服务重启都会丢。这是**有意的**，保持与现有记忆机制一致。未来若 MemorySaver 换成 Redis/SQLite 持久化，IntentTracker 也应同步迁移。

> 💡 **为什么加 `threading.Lock`？** Kafka Consumer 后台线程与 HTTP 请求线程可能并发访问同一 session 的轨迹，锁保证线程安全（与 `document_registry.py` 用 `RLock` 同理）。

#### ④ 模块级单例

```python
intent_agent = IntentAgent()      # 与 summary_agent = SummaryAgent() 同风格
intent_tracker = IntentTracker()
```

### 5.3 `app/services/rag_agent_service.py`（接入改造，改动最大）

共 5 处改动：

#### 改动 1：导入

```python
from app.agent.intent_agent import intent_agent, intent_tracker, IntentRecognitionResult

#### 改动 2：`_build_system_prompt` 支持 intent 注入

```python
def _build_system_prompt(self, intent_context: Optional[str] = None) -> str:
    base_prompt = dedent(""" ...原有系统提示... """).strip()
    # 意图段是「每轮临时」注入的，不作为独立消息持久化进 checkpointer
    if intent_context and intent_context.strip():
        base_prompt = f"{base_prompt}\n\n{intent_context}"
    return base_prompt
```

> 💡 **为什么把意图注入系统提示词，而不是单独存一条 SystemMessage？**
> - 单独存会**污染/堆积**对话历史（每轮一条），且 `trim_messages_middleware` 会保留 SystemMessage 导致历史膨胀。
> - 注入系统提示词是**每轮临时**的：本轮重建提示词 → 用完即弃 → 下轮重新识别重建。这与现有「每轮都新建 `SystemMessage(system_prompt)`」的行为一致，**不引入新的堆积问题**。

#### 改动 3：新增 `_recognize_intent` 辅助方法

```python
async def _recognize_intent(self, question, session_id) -> Optional[IntentRecognitionResult]:
    if not config.intent_recognition_enabled:
        return None                              # 开关关闭，零额外开销
    # 1. 最近 N 条消息（有界窗口）
    recent_messages = all_messages[-config.intent_recent_message_window:]
    # 2. 已有总结（更早对话的语义压缩）
    existing_summary, _ = SummaryAgent.extract_existing_summary(all_messages)
    # 3. 最近 2 条意图（判断 shift/依赖）
    previous_intents = intent_tracker.get_recent(session_id, k=2)
    # 4. 识别 + 记录
    result = await intent_agent.recognize(...)
    intent_tracker.record(session_id, result)
    return result
```

> 💡 **这就是「长上下文」策略的核心**：传给识别模型的只有 ① 当前问题 ② 最近 6 条 ③ 总结 ④ 最近 2 条意图——**全部有界**。即使对话 50 轮，识别输入也固定大小，不会变慢变贵。更早的语义信息靠 SummaryAgent 的总结保留。

#### 改动 4：`query` / `query_stream` 接入

两个方法在 `_summarize_and_update` 之后都加了：

```python
intent_result = await self._recognize_intent(question, session_id)
intent_context = intent_result.format_for_prompt() if intent_result else None
# query_stream 额外把意图先吐给前端：
if intent_result:
    yield {"type": "intent", "data": intent_result.to_dict()}
# 用本轮意图重建系统提示词（替换原先直接用 self.system_prompt）
system_prompt = self._build_system_prompt(intent_context)
messages = [SystemMessage(content=system_prompt), HumanMessage(content=question)]
```

> 💡 **流式里意图事件在内容流之前**：因为 `yield intent` 写在 `async for ... astream` 之前，前端会先收到 `{"type":"intent"}` 再收到 `{"type":"content"}`，可以先把"系统理解到的意图"展示出来。

#### 改动 5：`get_session_intents` + `clear_session` 联动

```python
def get_session_intents(self, session_id) -> list:    # 新增，供会话历史接口
    return [r.to_dict() for r in intent_tracker.get_all(session_id)]

def clear_session(self, session_id) -> bool:          # 追加一行
    self.checkpointer.delete_thread(session_id)
    intent_tracker.clear(session_id)                  # ← 新增：同步清空意图轨迹
```

## 六、关键设计决策汇总

| 决策点 | 选择 | 理由 |
|--------|------|------|
| **架构模式** | 复刻 SummaryAgent 辅助 Agent | 已验证、风格统一、改动集中 |
| **意图结果去向** | 注入系统提示词（临时） + 内存轨迹 | 不污染历史；轨迹供跨轮判断与展示 |
| **长上下文** | 有界输入（最近6条+总结+最近2意图） | 与对话总长无关，靠 summary 保留语义 |
| **多/复杂意图** | 识别+结构化+注入，**不自动编排** | 范围可控，与现有 Agent 架构一致（编排留给 AIOps Plan-Execute） |
| **结构化输出** | `with_structured_output` + JSON 兜底 + unknown 兜底 | 三级容错，最坏降级不崩 |
| **持久化** | 内存（与 MemorySaver 同生命周期） | 与现有记忆机制对齐 |
| **暴露方式** | SSE 意图事件 + 非流式 intent 字段 + 会话历史 intents | 前端可见，可观测 |
