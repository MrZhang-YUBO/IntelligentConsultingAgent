# 更新日志：向量知识库动态更新功能

> 版本：v1.4.0
> 日期：2026-06-17
> 功能：引入 **事件驱动 + 内容 Hash 检测 + 先删后增** 的向量知识库动态更新方案，支持文档增量更新与删除。

---

## 一、背景

### 原有方案（改造前）

上传文件后直接切分并入库（Milvus），存在以下问题：

| 问题 | 说明 |
|------|------|
| **重复索引（chunk 残留）** | 同一文件修改后重新上传，旧版本 chunk 不会被清理，导致向量库中同时存在新旧多版本内容 |
| **无法增量更新** | 缺少"文档"级别的关联，所有 chunk 都是孤立存在，无法定位哪些 chunk 属于哪篇文档 |
| **无法按文档删除** | 只能按文件名/文件路径模糊匹配删除，不可靠且不精准 |
| **同步阻塞** | 文件上传请求中同步执行"切分 + 向量化 + 入库"，大文件会阻塞 HTTP 响应 |
| **缺少变更检测** | 上传相同内容的文件仍会重复执行全流程，浪费算力 |

### 改造目标

1. **精准更新**：文档修改后，旧版本 chunk 被完整清理，只保留最新版本
2. **事件驱动**：HTTP 层快速返回，切分/入库由后台异步执行
3. **Hash 检测**：内容未变则跳过，避免重复计算
4. **可追踪**：每个 chunk 与 document_id 关联，删除/更新有据可查

## 二、整体架构

### 2.1 核心设计思想

```
┌──────────────────────────────────────────────────────────────────┐
│              Producer 侧（HTTP 请求中，同步 + 快速响应）              │
│                                                                  │
│  用户上传文件 → compute_file_hash() → document_registry 比对       │
│                  │                                                │
│                  ├─ 新文件：register_new → DOCUMENT_CREATED       │
│                  ├─ hash 相同：跳过（DOCUMENT_UNCHANGED）          │
│                  └─ hash 不同：DOCUMENT_UPDATED                    │
│                  │                                                │
│                  └──→  Kafka Topic: document_changes               │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                              │
                              ▼  Kafka 消息队列（解耦）
                              │
┌──────────────────────────────────────────────────────────────────┐
│             Consumer 侧（后台线程，异步处理）                         │
│                                                                  │
│  接收事件 →  ① 删除旧 chunk（按 document_id）                        │
│           │                                                     │
│           ├──② 重新切分文件（document_splitter_service）           │
│           │                                                     │
│           ├──③ 重新入库（vector_store_manager，带 document_id）    │
│           │                                                     │
│           └──④ 更新 registry（hash / chunk_count / status）         │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 2.2 完整数据流向图

```
HTTP /api/upload
    │
    ▼
保存到本地 → content_hash (SHA-256)
    │
    ▼
document_registry.is_content_changed()
    ├─ 新文件 (record=None)   → DOCUMENT_CREATED 事件
    ├─ hash 相同                → DOCUMENT_UNCHANGED（跳过）
    └─ hash 不同                → DOCUMENT_UPDATED 事件
            │
            ▼
    Kafka Topic: document_changes
            │
            ▼
    Consumer（FastAPI lifespan 中启动的后台线程）
            │
            ├─ DOCUMENT_CREATED: 切分 → 入库（带 document_id）
            │
            └─ DOCUMENT_UPDATED: 先删旧 chunk → 重新切分 → 重新入库
                    │
                    ▼
            Milvus collection: biz（含 document_id 独立字段）
                    │
                    ▼
            document_registry.update_hash() + update_status(ACTIVE)
                    │
                    ▼
            持久化到 data/document_registry.json
```

## 四、核心模块详解

### 4.1 `app/utils/hash_utils.py` — 内容 Hash 计算

**核心职责**：为每个文档生成唯一的内容指纹，用于检测变更。

**关键函数**：

| 函数 | 作用 |
|------|------|
| `compute_file_hash(file_path)` | 按 64KB 分块读取，计算 SHA-256，返回 `sha256:<hex>` |
| `compute_content_hash(text)` | 纯文本内容 hash（辅助函数） |
| `compute_metadata_hash(metadata)` | 结构化元数据 hash（`json.dumps(sort_keys=True)` 后 hash） |
| `hashes_equal(a, b)` | 安全比较（处理 None） |

**设计要点**：
- 使用 64KB 分块读取，大文件也不会 OOM
- 格式统一为 `sha256:<hex>`，便于未来替换算法
- 二进制读取，避免编码差异导致 hash 不一致

**使用示例**：
```python
from app.utils.hash_utils import compute_file_hash

h = compute_file_hash("./uploads/manual.pdf")
# 返回: "sha256:a1b2c3d4e5f6..."
```

### 4.2 `app/core/document_registry.py` — 文档注册表

**核心职责**：持久化维护 `document_id ↔ file_path ↔ content_hash` 的映射关系。

**存储格式**：`data/document_registry.json`（JSON 文件，原子写入：temp + rename）

**核心接口**：

| 方法 | 作用 |
|------|------|
| `find_by_path(path)` | 按文件路径查记录 |
| `find_by_id(doc_id)` | 按 document_id 查记录 |
| `is_content_changed(path, new_hash)` | 返回 `(bool, record)`，判断是否变更 |
| `register_new(path, name, hash)` | 新文档，生成 `doc_<8位随机字母数字>` 的 document_id |
| `update_hash(doc_id, new_hash)` | 更新 hash 与 chunk_count |
| `update_status(doc_id, status)` | 更新状态（PENDING / PROCESSING / ACTIVE / ERROR / DELETED） |
| `mark_deleted(doc_id)` | 软删除 |
| `list_all(include_deleted)` | 列出所有记录 |
| `stats()` | 返回统计信息 |

**数据模型 `DocumentRecord` 字段**：

```python
{
    "document_id": "doc_a1b2c3d4",        # 唯一 ID
    "file_path": "/uploads/ops-manual.pdf", # 归一化绝对路径
    "file_name": "ops-manual.pdf",         # 文件名
    "content_hash": "sha256:...",          # 当前内容 hash
    "chunk_count": 42,                     # 切分后 chunk 数
    "status": "active",                    # ACTIVE / PENDING / PROCESSING / ERROR / DELETED
    "created_at": 1718524800,
    "updated_at": 1718524900,
    "last_event_id": "evt_xxx",
    "error_message": null
}
```

**关键设计点**：
- `threading.RLock` 保护读写，多线程安全
- 临时文件写入后 `os.replace`，保证原子性，避免写入中断
- 文件路径归一化为 Posix 绝对路径，避免相对路径/绝对路径/斜杠差异导致重复记录

### 4.3 `app/core/kafka_client.py` — Kafka 事件客户端

**核心职责**：文档变更事件的生产者与消费者。

**消息格式（JSON）**：

```json
{
    "event_id": "evt_<12位随机>",
    "event_type": "DOCUMENT_CREATED" | "DOCUMENT_UPDATED" | "DOCUMENT_DELETED",
    "document_id": "doc_a1b2c3d4",
    "file_path": "/uploads/manual.pdf",
    "content_hash": "sha256:xxx",
    "old_hash": "sha256:yyy" | null,
    "timestamp": 1718524900
}
```

**生产者 `DocumentEventProducer`**：
- 默认使用 `confluent-kafka`，不可用时回退 `kafka-python`，再不可用使用内存队列（开发调试）
- `send(event)` 异步发送，`flush()` 确保发出，`close()` 关闭
- 内存模式下事件存入 `pending_events`，供 Consumer 轮询消费

**消费者 `DocumentEventConsumer`**：
- 后台线程 `_run_loop()` 轮询消费
- `set_handler(handler_func)` 设置事件处理回调
- `start()` / `stop()` 启停
- confluent-kafka → kafka-python → 内存模式 三层 fallback

**全局便捷函数**：
- `get_producer()` 获取全局单例
- `send_event(event)` 便捷发送

### 4.4 `app/core/milvus_client.py` — Milvus 客户端（schema 升级）

**核心改造**：`biz` collection schema 新增 `document_id` 独立字段（VARCHAR(100)），用于按文档批量删除。

**新的 collection schema**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | VARCHAR(100) | 主键，格式 `{document_id}_{chunk_index:04d}` |
| `document_id` | VARCHAR(100) | **新增**，用于按文档批量删除 |
| `vector` | FLOAT_VECTOR(1024) | 1024 维向量 |
| `content` | VARCHAR(8000) | chunk 文本 |
| `metadata` | JSON | 元数据（含 document_id / chunk_index 等） |

**自动 schema 升级逻辑**：`connect()` 中检测旧 collection 是否含 `document_id` 字段，如缺失或向量维度不匹配时自动删除重建。

### 4.5 `app/services/vector_store_manager.py` — 向量存储管理器

**核心改造**：chunk 与 document_id 关联 + 按 document_id 批量删除。

**chunk ID 规则**：
```
chunk_id = {document_id}_{chunk_index:04d}
例如: doc_a1b2c3d4_0000, doc_a1b2c3d4_0001, ...
```

**关键方法**：

| 方法 | 作用 |
|------|------|
| `add_documents_with_doc_id(docs, document_id)` | 带 document_id 入库，每个 chunk 的 metadata 中注入 `document_id` 和 `chunk_index` |
| `delete_by_document_id(document_id)` | 通过 Milvus `document_id == "{doc_id}"` 表达式批量删除；兜底走 metadata JSON 字段删除 |
| `add_documents(docs)` | 兼容旧代码，自动生成 `legacy_xxx` 的 document_id |
| `similarity_search(query, k)` | 向量相似度搜索 |

**删除策略**：
1. 首选通过独立 `document_id` 字段删除（高效，索引友好）
2. 兜底通过 `metadata["document_id"]` JSON 字段删除（兼容旧数据）

### 4.6 `app/services/vector_index_service.py` — 核心索引服务

**核心流程**（Producer 侧同步 + Consumer 侧异步）。

**Producer 侧**：`index_single_file(file_path, force=False)`

```
输入: 文件路径
   │
   ▼
1. compute_file_hash(file_path) → content_hash
   │
   ▼
2. document_registry.is_content_changed(path, hash)
   ├─ 未变更且非 force → 返回 DOCUMENT_UNCHANGED
   │
   ├─ 新文件 → register_new → 生成 document_id
   │                 │
   │                 └─ DOCUMENT_CREATED 事件
   │
   └─ 变更 → 用旧 document_id
                    │
                    └─ DOCUMENT_UPDATED 事件（含 old_hash）
   │
   ▼
3. send_event(event) → Kafka
   │
   ├─ 发送成功 → 返回（HTTP 快速返回）
   │
   └─ 发送失败 → 本地同步执行 _process_event_sync()（兜底）
```

**Consumer 侧**：`process_event(event)`

```
输入: DocumentChangeEvent
   │
   ▼
1. registry.update_status(PROCESSING)
   │
   ▼
2. event_type == DOCUMENT_DELETED → vector_store_manager.delete_by_document_id()
   │                                     → registry.mark_deleted()
   │
   ▼
3. event_type == DOCUMENT_UPDATED → vector_store_manager.delete_by_document_id()（先删旧）
   │
   ▼
4. document_splitter_service.split_file(file_path) → documents（重新切分）
   │
   ▼
5. vector_store_manager.add_documents_with_doc_id(documents, document_id)（重新入库）
   │
   ▼
6. registry.update_hash(new_hash, chunk_count) + update_status(ACTIVE)
```

**其他 API**：

| 方法 | 作用 |
|------|------|
| `index_directory(dir_path)` | 批量索引目录下所有支持文件 |
| `delete_document_by_id(document_id)` | 删除某文档（删 Milvus chunk + registry 标记 DELETED + 发删除事件） |
| `list_documents(include_deleted)` | 列出所有已索引文档（从 registry 读取） |

## 五、关键设计决策

### 5.1 事件驱动 vs 同步处理

| 维度 | 事件驱动（新方案） | 同步处理（旧方案） |
|------|-----------------|-------------------|
| HTTP 响应 | 快速（仅 hash 计算 + 发事件） | 慢（切分 + 向量化 + 入库全流程） |
| 可伸缩性 | Consumer 可独立部署/扩展 | 与 Web 服务器绑定 |
| 失败隔离 | Consumer 失败不影响 HTTP | 失败影响用户体验 |
| 复杂度 | 引入 Kafka + 注册表 | 简单直接 |

**结论**：事件驱动更适合知识库更新这类后台任务。

### 5.2 先删后增 vs 增量更新

| 策略 | 优点 | 缺点 |
|------|------|------|
| **先删后增（采用）** | 简单可靠，无版本残留，chunk_id 连续 | 短暂不可用窗口（毫秒级） |
| 增量（先增后删） | 无不可用窗口 | 需要比对新旧 chunk，复杂，chunk_id 不连续 |
| 版本化标记 | 可回滚 | 存储膨胀，查询复杂度高 |

**结论**：先删后增最简单可靠，不可用窗口极短（单篇文档毫秒级）。

### 5.3 chunk 关联设计：document_id 独立字段 vs metadata JSON

| 方式 | 查询性能 | 实现复杂度 |
|------|---------|----------|
| **document_id 独立字段（首选）** | 高，VARCHAR 索引友好 | 需 schema 升级 |
| metadata JSON 字段 | 中，JSON 路径查询 | 无需 schema 变更 |

**结论**：两种方式并存，独立字段为首选，JSON 为兜底。

### 5.4 持久化方式：JSON 文件 vs 数据库

| 方式 | 优点 | 缺点 |
|------|------|------|
| **JSON 文件（采用）** | 零依赖，易备份，原子写入 | 大规模（>10k 文档）性能下降 |
| SQLite / Redis | 高性能查询 | 增加依赖 |

**结论**：知识库文档量通常 < 10k，JSON 文件足够。未来规模大时可平滑迁移到 SQLite。

## 七、常见问题排查

### Q1: Kafka 不可用时怎么办？
**A**：kafka_client 内置三层 fallback：
1. confluent-kafka → kafka-python → 内存队列
Kafka 完全不可用时自动降级为内存模式，事件在内存队列中轮询消费，功能完整可用，只是失去分布式能力。

### Q2: Consumer 重启后会重复消费吗？
**A**：Kafka Consumer Group ID（`document-indexer-group`）使用 `enable.auto.commit=True`，由 Kafka Broker 维护 offset。正常情况下不会重复消费。极端场景下 `process_event()` 的"先删后增"是幂等的，重复消费不会导致数据不一致。

### Q3: registry 文件损坏怎么办？
**A**：
1. 停止应用，删除 `data/document_registry.json`
2. 重新启动后会创建空 registry
3. 重新索引目录（`POST /api/index_directory`）即可恢复
Milvus 中 document_id 字段仍然存在，registry 是辅助索引；记录丢失时，可通过 `metadata["document_id"]` 兜底删除。

### Q4: 旧版本 chunk 如何清理？
**A**：
- 旧版本 chunk（无 document_id 字段的旧数据）：
  1. Milvus collection schema 升级时会自动重建（如维度变更）
  2. 手动清理：可通过 `metadata["_source"]` 字段按文件路径删除

## 八、后续可扩展方向

| 方向 | 说明 |
|------|------|
| **增量 chunk 级更新** | 当前是"先删后增"，未来可改为"比对 chunk hash"，只变更的 chunk 才重新计算 |
| **文档版本历史** | registry 中保留历史版本，支持回滚到指定版本 |
| **并行消费** | 多 Consumer 实例并行处理，提升吞吐量 |
| **事件持久化到数据库** | registry 从 JSON 迁移到 SQLite / PostgreSQL |
| **增量索引进度 API** | 提供 `GET /api/indexing/status` 查看 Consumer 处理进度与状态 |
| **文件系统 watcher** | 监听 uploads 目录自动触发 reindex |
| **文档级 TTL** | 为文档设置过期时间，自动清理 |
| **更细粒度的并发控制** | 同一 document_id 的事件串行处理，避免并发更新冲突 |

## 九、回滚方案

如需回退到原有纯同步索引方案，只需：
