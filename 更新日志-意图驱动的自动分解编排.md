## 版本 2.0.0（2026-06-17）：架构优化 —— 子任务直接调工具 + 一次 LLM 汇总流式生成

### 2.0.0 核心动机

在 v1.x 中，每个子任务都调用 `rag_agent_service.query_stream`：即每个子任务都做了"意图识别 → 工具调用 → LLM 生成"完整链路。在有 3~5 个子任务时，会产生 **3~5 倍的 LLM 调用+延迟**，且用户的主体验其实是"最终回答"。v2 砍掉子任务的 LLM 调用，把它变成一个纯工具调用任务。

### 2.0.1 架构对比

| 维度 | v1.x | v2.0 |
|------|------|------|
| 子任务执行 | 完整 Agent 链路（意图识别+工具+LLM） | 仅调用工具（知识库检索/网络检索/时间/告警查询），无 LLM |
| 总 LLM 调用 | N + 1（N 个子任务 + 1 个汇总） | 1（只有汇总 LLM） |
| 子任务延迟 | N × 子任务 LLM 延迟 | max(t₁, t₂, …, t_N) — 工具调用本身比 LLM 快 1 个数量级 |
| 前端 | 显示 N 个子任务的流式内容区（DeepSeek 风格思考区） | 只显示一行进度提示，用户更专注于最终回答 |
| 前端内容区 | 有子任务独立流式内容区（可折叠/展开） | 无子任务内容区；汇总阶段直接流式输出最终回答 |

### 2.0.2 关键改动总览

| 文件 | 改动类型 | 核心变化 |
|------|----------|----------|
| `app/agent/orchestrator.py` | 重写 | 子任务改为 `_run_one_subtask_tool_only`：根据 `SubIntent.suggested_tools` 字符串（如 `retrieve_knowledge`/`web_search`/`get_current_time`/`query_prometheus_alerts`）直接 `asyncio.to_thread` 调用对应工具；汇总阶段用 `ChatQwen.astream` 一次 LLM 流式输出最终答案。不再产生 `orchestration_step` 与 `content{subtask_index=N}` 事件 |
| `static/app.js` | 修改 | 删掉 SSE 对 `orchestration_step` 与 `content{subtask_index}` 分支的处理；`initOrchestrationPanel` 改为"一行进度提示"（显示 `N` 个子问题数与各意图类型 + 建议工具）；`finishOrchestrationPanel` 简化为"更新进度行为 ✓ 已完成"，不再渲染子任务列表 |
| `static/styles.css` | 简化 | `.orchestration-steps` 强制 `display:none`；`.orchestration-panel` padding 更小，`.orchestration-title` 直接作为主进度行，不再包含子任务区块 |
| `app/services/rag_agent_service.py` | 无需改动 | 编排路径逻辑天然兼容：它只收集不带 `subtask_index` 的 `content`，在 v2 中这就是最终回答；fallback 行为保持不变 |
| `app/api/chat.py` | 无需改动 | 只透传 orchestrator 吐出的事件；orchestrator 不再产生子任务内容事件，自然就不会渲染 |

### 2.0.3 工具调度（核心新逻辑）

`orchestrator.py` 中的 `TOOL_REGISTRY` 把"意图识别阶段的建议工具字符串"映射到可调用对象：

```python
TOOL_REGISTRY: Dict[str, Any] = {
    "retrieve_knowledge":       retrieve_knowledge,       # 知识库 + 混合检索 + 重排
    "web_search":               web_search,               # Tavily 网络检索
    "get_current_time":         get_current_time,         # 当前时间（无参数）
    "query_prometheus_alerts":  query_prometheus_alerts,  # Prometheus 告警查询（无参数）
    # MCP 工具兜底：若意图识别输出这些字符串，则跳过编排（依赖主 Agent 兜底）
    "mcp_cls":         None,
    "mcp_monitor":     None,
}
```

子任务调用逻辑：
```
子任务 1 [意图=knowledge_search]
   suggested_tools = ["retrieve_knowledge"]
   → 调用 retrieve_knowledge("...")  → 文本片段
子任务 2 [意图=price_comparison]
   suggested_tools = ["retrieve_knowledge", "web_search"]
   → 并行调用两个工具
        retrieve_knowledge("...")  → 文本片段
        web_search("...")          → 文本片段
子任务 3 [意图=realtime_data]
   suggested_tools = ["query_prometheus_alerts", "get_current_time"]
   → 并行调用两个工具
```

### 2.0.4 事件协议（v2 更简洁）

- `orchestration_start`：告诉前端"正在并行检索 N 个子问题"（前端展示一行进度提示）
- `content`（无 `subtask_index`）：汇总 LLM 流式 token → 前端直接渲染为"最终回答"
- `orchestration_summary`：编排完成（模式 + 成功数/总数 + 总耗时），前端更新进度行状态

（v1.x 的 `orchestration_step` / `content{subtask_index=N}` 不再产生）

### 2.0.5 前端展示

- 进行中：`⚙ 正在分解为 N 个子问题并行检索 —— 涉及: [知识库](retrieve_knowledge)、[价格](retrieve_knowledge/web_search)、[告警](query_prometheus_alerts/get_current_time)…`（⚙ 有 1s 循环旋转动画）
- 完成后：`✓ 已完成并行检索（N₁/N 个子任务有结果），LLM 综合汇总生成回答`，下方显示"总耗时 X.Xs"

用户的主阅读流：**看一条简短进度 → 直接看到最终回答流式输出**（不再被子任务的中间内容干扰）。

| `static/app.js` | 修改 | 新增 `initOrchestrationPanel`（预先创建全部子任务条目、状态 pending）；新增 `_appendSubtaskContent`（把 `{subtask_index=N}` 的 `content` 流式追加到对应子任务的独立内容区）；运行中自动展开、完成后渲染为 Markdown、支持点击折叠/展开 |

### 1.1.2 事件协议（新增/变更）

| 事件 | 结构 | 前端行为 |
|------|------|----------|
| `orchestration_start` | `{type:"orchestration_start", data:{total:N, subtasks:[{index,question}]}}` | 一次性渲染全部子任务条目（○ 待开始） |
| `orchestration_step` | `{type:"orchestration_step", data:{step_index, status:"running"/"done"/"failed", subtask_question?, duration_ms?, error?}}` | 切换 icon / 文字 / 颜色；`running` 时自动展开内容区 |
| `content` (新增字段) | `{type:"content", data:"…", subtask_index?:number, node?:string}` | 带 `subtask_index` → 追加到对应子任务的内容区；**不带** → 渲染为主回答的流式内容（汇总阶段） |
| `orchestration_summary` | `{type:"orchestration_summary", data:{summary_mode, succeeded_count, total_count, elapsed_ms, fallback?:boolean}}` | 显示汇总模式与耗时；`fallback=true` 时前端回退到原 Agent 路径 |

### 1.1.3 为什么并行 + 流式

- **延迟显著降低**：N 个子任务从 `sum(T1..Tn)` 变为 `max(T1..Tn)`（典型场景下 3-5 倍性能改善）。
- **子任务流式不阻塞主回答**：每个子任务独立吐出 token，前端路由到各自的"子任务内容区"，**不会互相乱码、不会抢占主回答渲染**。
- **主回答也流式**：汇总阶段用 `astream` 直接把 LLM 的 token 持续推给前端，用户第一时间看到最终回答在"生长"，而不是等所有子任务完成后再一次性输出。

### 1.1.4 汇总阶段的回退策略（与 v1.0.0 一致，保留）

1. **LLM 综合汇总**（优先）：所有子任务成功时，用轻量模型（默认 qwen-turbo）把各子答案综合成一个流畅的最终回答。
2. **取唯一成功子任务结果**：只有 1 个子任务成功时，直接返回它的答案。
3. **拼接所有成功子任务结果**：≥2 个成功但 LLM 失败时，把各子答案原样拼接。
4. **回退到原 Agent**：全部子任务失败时，让 `rag_agent_service` 走原始 Agent 单轮路径兜底。

## 二、设计背景与动机

### 问题

之前的 Agent 在处理"多意图问题"或"复杂意图问题"时表现不佳：

- **多意图问题示例**："帮我介绍一下产品 A 的功能，再对比一下 A 和 B 的价格，最后给我一个采购建议" → 包含 3 个子意图。
- **复杂意图问题示例**："帮我写一份关于产品 A 的市场分析报告，包含市场规模、竞品对比、SWOT 分析和推荐策略" → 包含 4 个隐含的子问题。

现有 Agent 会将其当作一个大问题一次性处理，经常出现：

1. 检索不充分（一次检索只能覆盖部分子问题）
2. 生成回复冗长但重点不突出
3. 部分子问题被忽略或回答简略

### 解决方案

引入 **意图驱动的自动分解编排**（Intent-Driven Decomposition & Orchestration）：

```
用户多意图/复杂意图问题
       │
       ▼
 意图识别（已有能力）
       │
       ▼
 判定是否需要编排  ← 多意图数 ≥ 2 或 复杂意图标记
       │
       ├── 是 ▶ 进入编排路径
       │          ├─ 为每个子意图构造独立子问题
       │          ├─ 子问题 1  ──▶ rag_agent_service.query
       │          ├─ 子问题 2  ──▶ rag_agent_service.query
       │          └─ 子问题 N  ──▶ rag_agent_service.query
       │                      │
       │                      ▼
       │              用 LLM 汇总所有子答案
       │                      │
       │                      ▼
       │                返回一个最终回答
       │
       └── 否 ▶ 走原 Agent 路径（与之前完全一致）
```

### 为什么选择 rag_agent_service.query 作为子任务执行器

- **保持一致性**：每个子问题和普通用户问题走 **完全相同** 的处理链路（意图识别→检索→工具→生成），不会出现"编排路径能力与普通路径能力不一致"的问题。
- **避免代码重复**：不需要为编排路径再写一套 Agent，减少维护成本和 Bug 面。
- **可追踪性**：每个子任务的完整日志都保留在原服务中，便于事后排查。
- **非递归**：子任务被构造为单一意图的聚焦问题，即使再次走 `query` → 意图识别 → 编排判断，也 **不会** 再次触发编排（最低阈值 2 个子意图才触发），所以不会出现死循环。

## 三、核心新增文件详解：`app/agent/orchestrator.py`

### 3.1 数据结构

```python
class SubTask(BaseModel):
    intent_type: str                 # 此子任务对应的意图类型
    description: str                 # 子意图描述（用作 LLM 构造子问题的提示）
    subtask_question: str            # 构造出的独立子问题
    answer: Optional[str] = None     # 子任务执行后的原始答案（可能为空字符串）
    success: bool = False            # 此子任务是否成功（非空且非异常）
    error: Optional[str] = None      # 失败时的异常信息
    duration_ms: Optional[float] = None  # 单次执行耗时

class OrchestrationResult(BaseModel):
    triggered: bool = False          # 是否触发了编排（前端可据此决定是否展示进度）
    sub_tasks: List[SubTask] = []    # 子任务列表
    final_answer: Optional[str] = None
    summary_mode: Optional[str] = None  # 'llm_synthesis' | 'single_success' | 'concat_all' | 'fallback'
    elapsed_ms: Optional[float] = None
```

### 3.2 编排器关键方法

| 方法 | 职责 |
|------|------|
| `should_orchestrate(result)` | 根据意图结果判断是否进入编排（主开关 + 子意图数 + complex 标记） |
| `build_sub_tasks(result, original_question)` | 将每个子意图构造成独立可回答的子问题 |
| `_ask_llm_for_subtask(intent_type, description, original_question)` | 调用 LLM 生成更自然、聚焦的子问题（兜底：用 description 拼接） |
| `run_sub_tasks(tasks, session_id, enable_web_search, streaming)` | 串行执行子任务，每步产生 `orchestration_step` 事件 |
| `synthesize(tasks, original_question)` | 汇总子答案。4 种模式：LLM 综合 / 取唯一成功 / 拼接 / 回退 |
| `orchestrate(...)` | 主入口：以上步骤的组合，异步生成事件流 |

### 3.3 兜底机制（Fallback）

任何一步都可能失败，编排器内置 4 层兜底：

```
┌─ 模式 1：LLM 综合汇总（理想情况）
│  条件：≥ 2 个子任务成功 + LLM 调用成功
│  结果：由轻量模型（默认 qwen-turbo）将多个子答案融合为连贯回答
│
├─ 模式 2：取唯一成功子任务
│  条件：恰好 1 个子任务成功（其余失败/空）
│  结果：直接返回该子任务的答案
│
├─ 模式 3：拼接所有成功子任务答案
│  条件：≥ 2 个子任务成功但 LLM 汇总失败
│  结果：用简单的 markdown 标题分隔拼接
│
└─ 模式 4：回退到原 Agent
   条件：0 个子任务成功
   结果：orchestrate 返回 None（由 rag_agent_service 回退到原 Agent 路径）
```

## 四、配置项说明

在 `app/config.py` 中新增：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `intent_orchestration_enabled` | `True` | 总开关；关闭则 **永远** 不走编排路径 |
| `intent_orchestration_model` | `"qwen-turbo"` | 汇总用的轻量模型（子任务执行用主 Agent 模型，不是它） |
| `intent_orchestration_min_sub_intents` | `2` | 至少 N 个子意图才触发编排（避免小题大做） |

**主开关与最低阈值共同决定编排触发**，例如：

- 若 `min_sub_intents=2`，问题只有 1 个子意图 → 不编排
- 若意图被识别为 `is_complex=True` 且 描述中含有多个方面 → 即使只有 1 个"副意图"也可能触发（取决于实际配置）

## 五、调用链与流程

### 5.1 流式路径（主要使用路径）

```
用户发送问题
  │
  ▼
POST /chat_stream
  │
  ▼
rag_agent_service.query_stream(question, session_id)
  │
  ├── ▶ 对话历史总结压缩（既有能力）
  │
  ├── ▶ 多轮意图识别（既有能力，yield intent 事件）
  │
  └── ▶ 编排判断：should_orchestrate(result)
          │
          ├── 是：
          │     async for evt in orchestrator.orchestrate(...):
          │        ├── 每个子任务开始 → yield orchestration_step(status='running')
          │        ├── 每个子任务结束 → yield orchestration_step(status='done'/'failed')
          │        └── 最后汇总 → yield orchestration_summary
          │     (若 summary_mode != 'fallback') 切分 final_answer 流式吐出
          │     (若 summary_mode == 'fallback') 不产生最终答案，回退到下面的原路径
          │
          └── 否（或 fallback 后继续）：走原 Agent 流式
                  async for token, metadata in self.agent.astream(...):
                      yield content 事件
                  yield complete 事件
```

### 5.2 非流式路径（/quick_message）

逻辑相同，只是：

- 编排路径返回最终完整字符串
- 回退时调用 `self.agent.ainvoke` 一次性返回完整答案
- 由于非流式 API 不推流式事件，编排进度在日志中可见，但前端不展示进度面板（这是预期行为，非流式路径用户接受等待）

## 六、前端新增 UI 元素

当识别到多意图并触发编排时，AI 回复气泡顶部会额外显示一个 **编排进度面板**：

```
╔═════════════════════════════════════════════════════╗
║  ⚙ 正在分解为独立子问题并依次求解…                   ║
║  ────────────────────────────────────────────────   ║
║  ⏳  请分别介绍产品 A 和产品 B 的核心功能        求解中…  ║
║  ✓  请对比产品 A 和 B 的价格差异               完成(1.2s) ║
║  ✗  请基于功能与价格给出采购建议                 失败     ║
║  ────────────────────────────────────────────────   ║
║  📦 汇总模式: LLM 综合汇总 (2/3 个子任务成功)        ║
║     编排总耗时: 3.4s                                ║
╚═════════════════════════════════════════════════════╝

（下面紧跟 AI 的最终回答，与普通回复一致）
```

### 前端实现要点

- **`_ensureOrchestrationPanel`**：确保面板 DOM 存在并已插入到消息气泡内。
- **`updateOrchestrationPanel`**：每次收到 `orchestration_step` 事件时更新一个子任务的状态（图标 + 文本 + 颜色）。
- **`finishOrchestrationPanel`**：收到 `orchestration_summary` 时显示汇总模式与总耗时。
- **样式**：面板背景蓝紫渐变；运行中蓝色高亮；完成绿色；失败红色；汇总完成后面板切换为绿色边框。

## 七、潜在风险与注意事项

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| 子任务构造质量不稳定 | 子问题若偏离原意图，会影响最终汇总 | 使用结构化 JSON schema 约束 LLM 输出；失败时降级为"description 拼接" |
| 延迟叠加 | N 个子任务 × 单次 query 耗时 ≈ 5~30s 量级 | 仅在确实是多意图/复杂意图时触发；流式实时推送进度 |
| LLM 汇总质量不稳定 | 汇总答案可能不如人工整合自然 | 暴露 `summary_mode` 给前端，用户可看到是如何汇总的；必要时降级为拼接 |
| 循环编排 | 子任务再次调用 query，可能再次触发意图识别和编排判断 | ① 子问题被刻意构造为聚焦单一意图；② 最低阈值 ≥ 2 个子意图才触发；③ 理论上不会嵌套 |

## 八、测试建议

建议至少覆盖以下场景的人工回归测试：

| 场景 | 预期 |
|------|------|
| 单意图简单问题（"产品 A 是什么"） | 走原 Agent 路径，不出现编排面板 |
| 多意图问题（"介绍 A，对比 A 和 B 的价格，给采购建议"） | 触发编排，前端看到 N 个子任务进度，最终得到综合回答 |
| 复杂意图问题（"写一份产品 A 的市场分析报告，包含市场规模/竞品/SWOT/策略"） | 意图识别标记 complex → 触发编排 |
| 部分子任务失败 | 能观察到 failed 状态；最终仍能得到 fallback 的回答 |
| 关闭编排开关 `intent_orchestration_enabled=False` | 所有问题走原 Agent 路径，编排器不被调用 |
